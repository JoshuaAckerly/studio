# Asset Loading Guide

How the Noteleks game loads and displays sprite assets at runtime.

## Architecture Overview

```
manifest.json ──► LoadingScene ──► AssetManagerSimple ──► AnimationHelper
                   (2-stage)         (queue + load)        (create anims)
```

The pipeline uses a **two-stage loading** approach:

1. **Stage 1 (preload)** — LoadingScene fetches `manifest.json` from the server
2. **Stage 2 (create)** — The manifest drives image loading; once images complete, animations are created and the game starts

## Manifest Format

Generated by `tools/generate_noteleks_manifest.cjs`. Output lives at `public/games/noteleks/sprites/manifest.json`.

```json
{
  "frameSequences": {
    "/games/noteleks/sprites/Skeleton-Idle_": [
      "Skeleton-Idle_00.webp",
      "Skeleton-Idle_01.webp"
    ]
  },
  "sheets": { ... }
}
```

- **frameSequences**: Keyed by base URL prefix. Each value is an ordered array of filenames sorted by frame index.
- **sheets**: Spritesheet/atlas sidecar metadata (not used by the current WebP-only flow).

## Regenerating the Manifest

Run from the project root:

```bash
node tools/generate_noteleks_manifest.cjs
```

This scans `public/games/noteleks/sprites/` for `*.webp` files matching `Name_NN.webp`, groups them by prefix, sorts by index, and writes `manifest.json`. The `prebuild` npm script runs atlas conversion automatically before `vite build`.

**When to regenerate:**
- After adding, removing, or renaming sprite frames
- After exporting new animations from Spine

## Animation Config

Per-animation timing is defined in `LoadingScene.js`:

| Manifest Key | Alias | FPS | Repeat |
|---|---|---|---|
| `skeleton-idle` | `player-idle` | 8 | loop |
| `skeleton-run` | `player-run` | 12 | loop |
| `skeleton-walk` | `player-walk` | 12 | loop |
| `skeleton-attack1` | `player-attack` | 10 | once |
| `skeleton-attack2` | `player-attack` | 10 | once |
| `skeleton-jumpattack` | `player-jump` | 15 | once |

Animations not listed in the config default to 12 FPS, looping.

## Key Files

| File | Role |
|---|---|
| `scenes/LoadingScene.js` | Two-stage loader + progress UI |
| `utils/AssetManagerSimple.js` | Manifest parsing, image queuing, animation creation |
| `utils/AnimationHelper.js` | `createAnimation()`, `createAlias()`, `promoteFirstFrameTexture()` |
| `config/GameConfig.js` | Central config (screen, physics, player, enemies) |
| `tools/generate_noteleks_manifest.cjs` | Build-time manifest generator |

## Dev Mode vs Production

### Dev (`npm run dev`)

- Vite dev server on port **8084** with HMR
- Phaser loaded via CDN (`cdn.jsdelivr.net/npm/phaser@3.70.0`)
- `main-modular.js` waits up to 10s for `window.Phaser` to appear
- Console logging active (`app.debug = true`)
- Assets served from `public/` directory directly

### Production (`npm run build`)

- `prebuild` runs atlas conversion, then Vite produces optimized chunks
- Phaser chunk separated (`npm.phaser`) — not bundled into game code
- Console logging silenced in the Blade template
- Error tracking via `localStorage` + `sendBeacon` to `/__noteleks_error`

### Important Notes

- The game is **not SSR-rendered**. The route returns a plain Blade view, not an Inertia response. The `ssr` entry in `vite.config.js` is for other pages.
- Sprites are always loaded at runtime from `/games/noteleks/sprites/` — they are not bundled by Vite.
- The manifest must exist at deploy time. If missing, LoadingScene logs a warning and starts GameScene without assets.

## Adding a New Animation

1. Export frames as `AnimationName_00.webp`, `AnimationName_01.webp`, etc. into `public/games/noteleks/sprites/`
2. Run `node tools/generate_noteleks_manifest.cjs` to update `manifest.json`
3. If the animation needs custom timing, add an entry to `ANIM_CONFIG` in `LoadingScene.js`
4. If the animation needs a player-facing alias, add it to `ALIAS_MAP` in `utils/AnimationHelper.js`
5. Reference the alias key (e.g. `player-newmove`) in your game code
